## 1. 目標（Goal）

### 1-1. 最終目標

| 項目 | 内容 |
|------|------|
| 達成期限 | 60歳時（約7年後） |
| 目標収入 | 月25万円以上（自動売買の利益のみ） |
| 収入の性質 | 労働収入に依存しない継続的・安定的な不労所得 |
| 生活費の目安 | 月25万円（現在の生活水準を維持できる最低ライン） |

### 1-2. 中間目標（マイルストーン）

| 期間 | 資産目標 | 月収換算（4%引出） | 主な達成条件 |
|------|---------|-----------------|------------|
| 1年後 | 120万円 | 4,000円/月 | バックテスト完成・戦略確立 |
| 3年後 | 200万円 | 6,700円/月 | 実運用開始・年利15%以上安定 |
| 5年後 | 400万円 | 13,300円/月 | 年利20%以上・複数戦略並走 |
| 60歳時（退職金投入後） | 800万円以上 | 26,700円/月 | 退職金300万追加投入・月25万達成 |

> ⚠️ **注意**：年利20%継続はプロでも困難な水準。退職金（300万円）の投入が月25万達成の鍵となる。退職金は絶対に失わない運用規律を守ること。

## 2. 条件（Constraints）

### 2-1. 資金条件

| 項目 | 内容 |
|------|------|
| 現在の運用資金 | 100万円 |
| 月次積立 | 1万円/月 |
| 退職金（60歳時） | 300万円（予定） |
| 最大リスク/トレード | 運用資金の2%以内（＝2万円/トレード） |
| 退職金の扱い | 実運用で年利15%以上が3年安定した後のみ投入可 |

### 2-2. 売買条件

| 項目 | 内容 |
|------|------|
| 対象市場 | 日本株（東証プライム）現物のみ |
| 保有期間 | 1〜10日（スイングトレード）10日目後場引けに強制決済 |
| レバレッジ | なし（現物のみ・資金以上の投資は行わない） |
| 空売り | 不可（現物買いのみ） |
| FX・先物 | 対象外 |
| 信用取引 | 対象外 |
| 1銘柄最大投資額 | 運用資金の30%以内（過集中防止） |
| 同時保有銘柄数 | 最大3〜5銘柄 |

### 2-3. リスク管理条件（絶対ルール）

> 💡 以下のルールは感情や状況に関わらず機械的に遵守する。違反した場合はその月の取引を停止し原因を記録する。

| ルール | 内容 |
|--------|------|
| 損切りライン | 買値からATR×2（最大-5%）で必ず損切り。例外なし |
| 月次最大損失 | 運用資金の10%超で当月の新規取引停止 |
| 強制決済 | 保有10日目の後場引けで理由に関わらず決済 |
| 決算週 | エントリー禁止（決算発表前後3営業日は除外） |
| 退職金投入条件 | 実運用年利15%以上×3年継続を確認後のみ |

## 3. 手段（Means）

### 3-1. 使用ツール・サービス

| ツール | 内容 |
|--------|------|
| データソース | J-Quants API（Standardプラン）- 東証JPX公式データ・10年分の株価/財務情報 |
| AI開発支援 | Claude Pro + Claude Code - コード生成・バグ修正・戦略設計に活用 |
| 開発言語 | Python（Claude Codeが生成・管理） |
| バージョン管理 | Git（GitHub）- 戦略の変更履歴・バックテスト結果を記録 |
| 実行環境 | Windows PC（ローカル） |
| バックテストライブラリ | vectorbt または backtrader |
| データDB | SQLite（ローカル保存） |
| 将来：自動発注 | SBI証券 kabuステーションAPI（実運用移行時） |

### 3-2. 取引戦略（バックテスト対象）

#### 戦略A：トレンドフォロー型（メイン戦略）

| 項目 | 条件 |
|------|------|
| 銘柄ユニバース | 東証プライム・株価500〜3,000円・出来高50万株/日以上・時価総額500億以上 |
| エントリー条件 | ①株価が25日MA上　②5日MAが25日MAをゴールデンクロス　③ADX>25　④RSI 40〜65　⑤出来高20日平均の1.2倍以上 |
| 決算除外 | 決算発表前後3営業日はエントリー禁止 |
| 損切り | 買値 - ATR(14)×2（最大-5%） |
| 利確 | 買値 + ATR(14)×4（リスクリワード1:2） |
| 強制決済 | 保有10日目の後場引け |

#### 戦略B：出来高ブレイクアウト型（サブ戦略）

| 項目 | 条件 |
|------|------|
| エントリー条件 | ①52週高値更新　②出来高が20日平均の2倍以上　③株価500〜3,000円　④時価総額500億以上 |
| 損切り | ブレイク起点の安値 or -3%（小さい方） |
| 利確 | +8% or 保有10日目の後場引け |

#### 戦略C：財務スクリーニング＋モメンタム型（J-Quants財務データ活用）

| 項目 | 条件 |
|------|------|
| 財務条件 | PBR<1.5 かつ ROE>10%（J-Quants財務情報より） |
| モメンタム | 過去3ヶ月リターンが上位20%銘柄 |
| 損切り | -5% |
| 利確 | +10% or 保有10日目の後場引け |

## 4. 手順（Procedure）

### 4-1. フェーズ全体像

| フェーズ | 期間 | 内容 | 達成判定基準 |
|---------|------|------|------------|
| Phase 0 | 〜1ヶ月 | 環境構築・データ基盤整備 | J-QuantsからDB保存まで動作確認 |
| Phase 1 | 1〜3ヶ月 | バックテスト開発・戦略検証 | 3戦略のバックテスト完成・結果記録 |
| Phase 2 | 3〜6ヶ月 | ペーパートレード（手動模擬） | 3ヶ月の模擬取引記録・勝率確認 |
| Phase 3 | 6ヶ月〜1年 | 少額実運用（50万以下） | 6ヶ月連続で月次損失10%以内 |
| Phase 4 | 1〜3年 | 実運用フル稼働・自動化 | 年利15%以上を1年以上継続 |
| Phase 5 | 3年〜退職 | 退職金投入・最終目標達成 | 月25万円以上を3ヶ月連続達成 |

### 4-2. Phase 0：環境構築（最初の1ヶ月）

#### Step 1：プロジェクト初期化

- Windowsにて作業フォルダ作成：`swing-trade-system/`
- Git初期化（`git init`）・GitHubリポジトリ作成
- `.env` ファイル作成（J-Quants APIキーを記載）
- `.gitignore` に `.env` を追加（APIキーをGitに上げない）
- Python仮想環境作成（`.venv`）

#### Step 2：ライブラリのインストール（Claude Codeに依頼）

Claude Codeへの指示例：

```
「Windows環境にPython仮想環境を作成し、
pandas / numpy / requests / python-dotenv /
vectorbt / matplotlib / japanize-matplotlib /
sqlalchemy / schedule をインストールする
requirements.txtとセットアップ手順を作成してください」
```

#### Step 3：J-Quants APIデータ取得・DB保存（Claude Codeに依頼）

- 東証プライム全銘柄の過去5年分の日次株価をSQLiteに保存
- 毎日の差分更新スクリプトも作成
- Windowsタスクスケジューラで平日15:45に自動実行設定

### 4-3. Phase 1：バックテスト開発（1〜3ヶ月）

#### 進め方

1. 戦略Aのシグナル生成コードをClaude Codeに作成させる
2. 過去5年分でバックテスト実行
3. 結果をスプレッドシートに記録（下記評価指標）
4. パラメータを変えて再テスト（過剰最適化に注意）
5. 戦略B・Cも同様に実施
6. 3戦略の中から最も安定した戦略を選定

#### バックテスト評価指標（合格基準）

| 指標 | 合格ライン | 意味 |
|------|----------|------|
| 年利（CAGR） | 15%以上 | 複利年間成長率 |
| 最大ドローダウン | -25%以内 | 最大の資産下落幅 |
| 勝率 | 40%以上 | 利益トレードの割合 |
| プロフィットファクター | 1.3以上 | 総利益÷総損失 |
| シャープレシオ | 0.8以上 | リスク調整後リターン |
| トレード回数 | 年50回以上 | 統計的に有意な回数 |

> ⚠️ **過剰最適化（カーブフィッティング）に注意**：バックテストが良すぎる場合（年利50%超など）は過剰最適化の疑いが強い。直近1〜2年のデータを「アウトオブサンプル」として残し、学習に使わずに最後に検証することを必ず実施すること。

### 4-4. Phase 2：ペーパートレード（3〜6ヶ月）

- バックテストで合格した戦略でリアルタイムシグナル生成スクリプトを作成
- 毎日15:30頃にシグナルを確認し、Excelに手動記録
- 実際には売買しないが、シグナル通りに動いた場合の損益を3ヶ月間記録
- **目標**：3ヶ月の模擬取引で月次損失が運用資金の10%以下×3回
- 達成できない場合はPhase 1に戻り戦略を見直す

### 4-5. Phase 3以降：実運用・自動化

- Phase 2合格後、まず50万円以下で実運用開始
- SBI証券のkabuステーションAPIとの接続はClaude Codeに実装依頼
- 6ヶ月間で年利換算15%以上かつ月次損失10%以内を継続できた場合、残り50万を追加
- 退職金（300万円）の投入は実運用で年利15%以上×3年継続後のみ実施

## 5. プロジェクト構成・Claude Code活用方針

### 5-1. フォルダ構成

```
swing-trade-system/
├── data/
│   ├── fetch_prices.py       # J-Quants: 株価取得
│   ├── fetch_financials.py   # J-Quants: 財務情報取得
│   ├── update_daily.py       # 毎日差分更新
│   └── universe.py           # 銘柄ユニバース管理
├── strategy/
│   ├── signals_a.py          # 戦略A シグナル生成
│   ├── signals_b.py          # 戦略B シグナル生成
│   ├── signals_c.py          # 戦略C シグナル生成
│   ├── filters.py            # 共通フィルター
│   └── position_size.py      # 資金管理ロジック
├── backtest/
│   ├── engine.py             # バックテストエンジン
│   └── report.py             # 結果レポート生成
├── live/
│   ├── scanner.py            # 毎日のシグナルスキャン
│   └── order.py              # 発注ロジック（Phase3以降）
├── logs/
│   └── trade_log.xlsx        # トレード記録
├── .env                      # APIキー（Git管理外）
├── .gitignore
└── requirements.txt
```

### 5-2. Claude Codeへの指示パターン

**データ取得系**

```
「J-Quants APIのAPIキーは.envのJQUANTS_API_KEYに設定済み。
東証プライム全銘柄の日次株価（OHLCV）を取得し、
SQLiteのprices.dbに保存するdata/fetch_prices.pyを作成してください。
既存データがある場合は差分のみ取得してください。」
```

**バックテスト系**

```
「strategy/signals_a.pyを使って、過去5年分のバックテストを
backtest/engine.pyで実行し、年利・最大ドローダウン・勝率・
プロフィットファクター・シャープレシオを
backtest/report.pyでグラフ付きで出力するコードを作成してください。」
```

### 5-3. Claude Codeに任せること・自分が判断すること

| Claude Codeに任せる | 自分が判断・決定する |
|--------------------|-------------------|
| APIラッパーの実装 | 戦略パラメータの最終決定 |
| バックテストエンジンの実装 | バックテスト結果の解釈・戦略選定 |
| データ処理・クリーニング | 実運用移行の判断タイミング |
| グラフ・レポート生成 | 損切りルールの遵守（機械的に実行） |
| バグ修正・リファクタリング | 退職金投入の最終判断 |
| スケジューラ設定コード | 月次の振り返りと改善方針決定 |

## 6. 達成確率を上げる仕組み

### 6-1. 継続できる仕組み（行動設計）

| タイミング | 実施内容 |
|-----------|---------|
| 毎営業日15:30 | シグナルスクリプト実行・翌日候補銘柄を確認（5分） |
| 毎週末（土曜） | 週次損益集計・トレード記録更新（30分） |
| 毎月末 | 月次レポート作成・目標との乖離確認・翌月方針決定（1時間） |
| 四半期末 | バックテスト結果の再検証・戦略パラメータ見直し（半日） |
| 年1回 | 本ドキュメントの見直し・目標・条件・手順の更新 |

### 6-2. 失敗パターンと対策

| 失敗パターン | リスクレベル | 対策 |
|------------|-----------|------|
| 損切りできず大損 | 🔴 最高 | システムによる自動損切り設定・感情排除 |
| 過剰最適化した戦略を実運用 | 🔴 高 | アウトオブサンプル検証を必須とする |
| 退職金を投機に使う | 🔴 最高 | 投入条件（3年15%以上）を文書化し厳守 |
| 途中で飽きて継続できない | 🟡 中 | 毎日5分の習慣化・月次記録でモチベ維持 |
| 下落相場で無理にエントリー | 🔴 高 | 25日MA・ADXフィルターで自動的に除外 |
| システムエラーで誤発注 | 🟡 中 | Phase3まで手動確認・ログ必須化 |

### 6-3. 現実的な達成確率の評価

| シナリオ | 確率 | 条件 |
|---------|------|------|
| 月25万達成（60歳時） | 約30% | 年利20%以上を5年継続＋退職金投入 |
| 月15万前後達成 | 約50% | 年利15%程度を継続 |
| 元本割れなしで継続 | 約70% | リスク管理ルール遵守が前提 |
| 退職金を失うリスク | 約5% | 投入条件を守れば大幅に低減可能 |

> 💡 **達成確率を上げる最大のレバレッジ**：「損切りルールの厳守」と「退職金の慎重な投入判断」の2点に尽きる。システムを高度化するより、この2点を守る規律の方が長期的なリターンへの影響が大きい。
